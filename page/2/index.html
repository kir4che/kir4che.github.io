屬性 Props

父組件 → 子組件傳遞之屬性，用來設定組件初始化的相關資料，可讀不可寫。

PropTypes

children

JSX element 的子 element，調用包在標籤中的內容。

  • React.FC

    已在 FC 定義好 children 型別、函式回傳值

    💡不可使用 setState()，須改用 useState()、useEffect() 等。

  • 直接定義

    • JSX.Element（不支持 string）
    • ReactChild
    • ReactNode

狀態 State

維護組件內的資料，用於會隨時間而變動之事件，為私有屬性。

初始化

於 constructor 定義初始 state

💡JS this 取決於當下誰執行 function

setState

setState(updater, [callback])

改變組件 state，並重新調用 render 以渲染。

  • 接受 object、function 作為參數
  • 不會馬上更新，須利用利用上一次結果 ➪ prevState

💡不要寫在 render() ➪ 死迴圈

生命週期

Mounting

建立組件時被執行

componentDidMount() 於組件被 mounted 後執行 ➪ 啟動

Updating

更新組件時執行

componentDidUpdate() 於 re-rending 更新組件後執行

Unmounting

當組件將要從 DOM 刪除時觸發

componentWillUnmount() 於組件被刪除前執行 ➪ 清除

  4. 子組件功能偵測父組件中的 function

  6. 觸發渲染機制

  7. 重新渲染 App 所有組件

  1. 定義父、子組件之 Props、State type

  2. 父、子組件 constructor 初始化

  3. super(props)

  4. this.state

  5. 調用 setState

  6. 記得保證 this 永遠指向 APP，不管是誰執行它。

    this.function = this.function.bind(this)

[React] Hooks vs Render Props vs HOC

2022-06-25

組件封裝 ➪ 重複使用程式碼

Higher Order Component（HOC）高階組件

抽離程式碼間的共用邏輯，把 component 作為參數傳入以重複使用之函式。

Pros

  • 寫法簡潔

Cons

  • 看不到接收、回傳之結構 ➪ 可讀性差
  • 不能確定多個 HOC 之順序，且不知 props 從何而來。
  • 同名 props ➪ 命名衝突（不會發出警告）
  • 靜態組合 ➪ 不能用於 render、lifeCycle

Example

Hooks

可在不用 class 情況下，藉助 Hooks 在純函數位件中使用狀態和其他 React 功能。

💡不能於條件式、迴圈及一般的 JS 函式中呼叫 Hooks

Pros

  • 可對 props 重命名
  • 無限制資料獲取及使用範圍
  • 在 class 不能用

Cons

  • 無法實現 Component 之間共享程式碼

Example

Render Props

用一值為函式的 prop 來在 React Component 之間共享程式碼

Pros

  • 可對 props 重命名
  • 能夠動態決定 render 什麼
  • 不會產生沒用的空組件 ↔ HOC

Cons

  • 程式碼冗長
  • 難以使用 render 以外之資料 ➪ 巢狀地獄

缺點

  • 生成的 className 是隨機字串 ➪ 除錯不易
  • 預設無 CSS 語法高亮、提示

安裝

  • npm install --save styled-components
  • yarn add styled-components

Common inventory policies

  • the（Q, R）policy：I < R（= reorder point）則訂 Q 個貨 ➪ 最小化存貨

    • optimal R

      • the expected total cost for the future, need to estimate / forecast / predict future demands.

  • the（s, S）policy：當你的存貨 I < s 則訂 S
